# S3 Buckets

`podman:s3-setup` creates the S3 buckets your application needs and applies a CORS policy to the ones that serve requests directly to a browser (signed URLs, public assets, etc.). It talks to S3 directly over the API — no `aws` CLI or extra container required.

## Requirement: the AWS SDK

The command needs [`aws/aws-sdk-php`](https://github.com/aws/aws-sdk-php), but the package doesn't require it by default — most apps that use `laravel-podman` don't need S3 at all, and the SDK is a sizeable dependency. Install it yourself when you do:

```bash
composer require aws/aws-sdk-php
```

Running `podman:s3-setup` without it installed prints a friendly error instead of failing with a class-not-found crash:

```
The AWS SDK is required to use this command. Install it with "composer require aws/aws-sdk-php".
```

Every other command in the package is unaffected either way — the check only runs when `podman:s3-setup` itself is invoked.

## Credentials

The command reuses whatever is already configured on your app's `s3` disk in `config/filesystems.php` (`key`, `secret`, `region`, `endpoint`, `use_path_style_endpoint`) — typically your `.env`'s `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`/`AWS_ENDPOINT` and friends. There's no separate credential config to keep in sync.

This works against AWS S3 or any S3-compatible service that supports signed URLs (MinIO, RustFS, Garage, etc.).

## Configuring buckets

Two config keys, publishable with `php artisan vendor:publish --tag="podman-config"`:

```php
's3_buckets' => env('PODMAN_S3_BUCKETS', [
    'local',
    'conversions',
    'segments',
    'secrets',
]),

's3_cors_buckets' => env('PODMAN_S3_CORS_BUCKETS', [
    'conversions',
    'segments',
    'secrets',
]),
```

- `s3_buckets` — every bucket `podman:s3-setup` creates. Creation is idempotent; a bucket that already exists is treated as success, not a failure.
- `s3_cors_buckets` — which of those buckets also get the CORS policy applied. This must be a subset of `s3_buckets`, or the command will fail trying to apply CORS to a bucket that was never created.

Both accept either a plain PHP array or a comma-separated string (handy for the env variable form).

## The CORS policy

The policy applied to `s3_cors_buckets` comes from the `s3` preset's `cors.json` — resolved at `containers/stubs/s3/cors.json` if you've published it, otherwise the vendor-provided one at `vendor/foxws/laravel-podman/stubs/s3/cors.json`. Unlike the other presets, `s3` has no `quadlets/`/`runtimes/` split and never goes through `podman:generate` — `podman:s3-setup` reads the file directly. Publish it before running the setup command if you want to customize it:

```bash
php artisan podman:publish s3
php artisan podman:s3-setup
```

Edit `containers/stubs/s3/cors.json` to customize the policy for your app; it's plain [S3 CORS configuration JSON](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cors.html):

```json
{
    "CORSRules": [
        {
            "AllowedOrigins": ["*"],
            "AllowedMethods": ["GET", "HEAD"],
            "AllowedHeaders": ["*"],
            "ExposeHeaders": ["Content-Length", "Content-Range", "Accept-Ranges", "ETag"],
            "MaxAgeSeconds": 7200
        }
    ]
}
```

Browsers enforce CORS even on signed URL requests, so any bucket a `<img>`/`<video>`/`fetch()` call reads from directly needs this — buckets only ever touched server-side (e.g. via a queued job) don't.

## Usage

```bash
php artisan podman:s3-setup
```

This creates every bucket in `s3_buckets`, then applies the CORS policy to every bucket in `s3_cors_buckets`. Output reports each step and, on failure, which buckets specifically failed:

```
Creating buckets...
  -> local
  -> conversions
  -> segments
  -> secrets
Applying CORS policy...
  -> conversions
  -> segments
  -> secrets
Done.
```
