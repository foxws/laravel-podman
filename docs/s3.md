# S3 Buckets

`podman:s3-setup` creates the S3 buckets your app needs and applies a CORS policy to buckets browsers access directly (signed URLs, public assets, and similar). It talks to S3 over the API, so you do not need the `aws` CLI or another container.

## Requirement: the AWS SDK

This command needs [`aws/aws-sdk-php`](https://github.com/aws/aws-sdk-php), but the package does not require it by default because many apps do not use S3 and the SDK is large. Install it when needed:

```bash
composer require aws/aws-sdk-php
```

If you run `podman:s3-setup` without the SDK, you get a clear error instead of a class-not-found crash:

```
The AWS SDK is required to use this command. Install it with "composer require aws/aws-sdk-php".
```

Other package commands are unaffected. This check only runs for `podman:s3-setup`.

## Credentials

The command reuses your app's existing `s3` disk config in `config/filesystems.php` (`key`, `secret`, `region`, `endpoint`, `use_path_style_endpoint`) — usually values from `.env` such as `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and `AWS_ENDPOINT`. There is no second credential config to maintain.

It works with AWS S3 and S3-compatible services that support signed URLs (MinIO, RustFS, Garage, and others).

## Configuring buckets

Two config keys, publishable with `php artisan vendor:publish --tag="podman-config"`:

```php
's3_buckets' => env('PODMAN_S3_BUCKETS', [
    'local',
    'conversions',
    'segments',
    'secrets',
]),

's3_cors_buckets' => env('PODMAN_S3_CORS_BUCKETS', [
    'conversions',
    'segments',
    'secrets',
]),
```

- `s3_buckets` — every bucket `podman:s3-setup` creates. Creation is idempotent, so existing buckets are treated as success.
- `s3_cors_buckets` — which of those buckets also get the CORS policy. This must be a subset of `s3_buckets`, or CORS apply will fail for missing buckets.

Both accept either a plain PHP array or a comma-separated string (handy for the env variable form).

## The CORS policy

The CORS policy for `s3_cors_buckets` comes from the `s3` preset `cors.json`: `containers/stubs/s3/cors.json` if published, otherwise `vendor/foxws/laravel-podman/stubs/s3/cors.json`. Unlike other presets, `s3` has no `quadlets/`/`runtimes/` split and does not use `podman:generate`; `podman:s3-setup` reads this file directly. Publish first if you want to customize it:

```bash
php artisan podman:publish s3
php artisan podman:s3-setup
```

Edit `containers/stubs/s3/cors.json` to customize your policy. It is standard [S3 CORS configuration JSON](https://docs.aws.amazon.com/AmazonS3/latest/userguide/cors.html):

```json
{
    "CORSRules": [
        {
            "AllowedOrigins": ["*"],
            "AllowedMethods": ["GET", "HEAD"],
            "AllowedHeaders": ["*"],
            "ExposeHeaders": ["Content-Length", "Content-Range", "Accept-Ranges", "ETag"],
            "MaxAgeSeconds": 7200
        }
    ]
}
```

Browsers enforce CORS even for signed URLs, so any bucket read directly by `<img>`, `<video>`, or `fetch()` needs this policy. Buckets used only server-side (for example from queued jobs) usually do not.

## Usage

```bash
php artisan podman:s3-setup
```

This creates all buckets in `s3_buckets`, then applies CORS to buckets in `s3_cors_buckets`. Output shows each step and which buckets failed, if any:

```
Creating buckets...
  -> local
  -> conversions
  -> segments
  -> secrets
Applying CORS policy...
  -> conversions
  -> segments
  -> secrets
Done.
```
